Authoring Guide

Contributing

At this time, contributions for new content are not being accepted.

Site Source

This site is built using Hugo and follows the directory structure outlined by that tool.

The following are the most important directories for authors:

  • content - This directory houses the main content and forms the base directory of the rendered site.
  • data - This directory houses data files used for site content.

Content Creation

Document Structure

Documents are organized using the Page Bundles feature of Hugo, meaning that each document is organized into a dedicated directory. For more complex tiered documents, there may be sub-directories which represent branches or leaves of the document structure. The guides content directory provides an example of this structure. When examining this example, you’ll notice that each directory contains either an “_index.md” or an “index.md” file depending on whether this directory is a branch or a leaf. Branches vs Leaves are defined as follows:

  • Branches are those directories which contain content sub-directories.
  • Leaves are those directories which do not contain any other content sub-directories. Leaves may contain other directories (such as an images directory) which contain content specific to that particular piece of content.

The index files within a directory represent the main content file. These files will contain a Font Matter header which defines certain attributes used for rendering the page. The Font Matter is described below.

Font Matter

Font Matter is used to define attributes of a page. An example is below. Note that the dashes at the start and end are required since they delineate the Font Matter from the rest of the content.

---
title: "Page Title"    
summary: "A summary to be displayed within index listings."
author: dspinhirne
contributors:
  - johndoe
  - "Jane Doe"
date: 2019-01-31
lastmod:2019-02-31
notoc: false
nofooter: false
weight: 100
menu:
  main:
    identifier: "path_to_this_page-title"
    parent: "path_to_this"
    weight: 100
---

YAML is the preferred format for Font Matter. The fields are as follows:

  • title - The page title. This will be displayed in the header of the rendered page.
  • summary - A summary which will be displayed within index listings. If this is not provided, then one will be created from the first portions of the content. It is generally better to manually provide a summary.
  • author - Either a string value or a key from site data.
  • contributors - a list of individuals who contributed significantly to the content but aren’t direct authors. Provides a means of crediting people for their help. Follows the same format as authors.
  • date - The date the document was first published. Use the format YYYY-MM-DD.
  • lastmod - The date the document was updated.
  • notoc - if set true, then no table of contents will be generated for the document.
  • nofooter - if set true, then no footer will be generated for the document.
  • weight - used for ordering pages within index listings as well as within printable views of structured documents.
  • menu - used to insert the page into the main menu. More details below.

In addition to these, there are other keys which are used within specific types of pages. These are as follows:

  • cascade - this field serves to propagate attributes down to child pages from a parent page.
  • heading - used to provide a main heading to pages. see the _index.md of one of the guides for an example.
  • subheading - used to provide a sub-heading to pages. see the _index.md of a howto branch of a guide for an example.
  • noprintable - used to exclude a leaf/branch from a higher-order printable view. see the hcx guide as an example where the HowTos have been excluded from the main printable view.

Page content is to be authored in markdown format. There is a markdown editor provided within this site which may be used to test out your formatting. For those who are new to markdown, it is recommended to look at the source for an existing document as an example.

It is very common that a link to a particular URL is used in multiple places. For very commonly referenced URLs, it is better to utilize the urlfor shortcode (see below) which will enable the URL to be managed centrally within site data. In addition to enabling the URL to be managed from a single location, the other advantage to this approach is that the URL will be displayed within the Information page of this site.

Menus

Menus are Hugo feature which are being used within this site. Only the “main” menu is actually being used. The fields of a menu entry are defined below. It is best to look at a existing page for an example of a proper menu entry.

  • identifier - A unique key for this entry. The convention is to prefix the ancestor keys, separated by underscores, ending with the page title (replacing spaces with “-” and omitting special characters). Keys should be all lower case.
  • name - Typically omitted. The page title will be used by default. If the page title is really long, then manually specify a shorter name.
  • parent - The parent key. This entry will be inserted under the specified parent. Top-level entries will have no parent.
  • weight - Used to order the entry within its parent. Lower values are listed first.

Shortcodes

Hugo provides access to Shortcodes within content files which help to provide additional functionality to standard markdown. Shortcodes are similar to functions within programming languages in that they accept arguments which dictate their behavior. Shortcodes are inserted into a document using the format: {{< shortcodename parameters >}} . Some Shortcodes are inline while others may contain content. Those which contain content will require a closing tag at the end of the Shortcode which is simply the Shortcode name with a slash prepended. An example of this is:

{{< shortcode >}}
*content...* 
{{< /shortcode >}}

A list of standard Shortcodes is available from the Hugo documentation. Of particular value are the following Shortcodes:

  • ref - generates a reference to a page within this site. generates a build error for broken references. a nice safe way to cross-reference pages.
  • highlight - provides code highlighting
  • param - extracts a parameter from Font Matter

Custom Shortcodes

In addition to the standard Shortcodes defined by Hugo, the following custom Shortcodes have been created for this site.

contributor

Generates a link to an author. Accepts a single parameter representing the author’s key within site data.

Example
{{< contributor "dspinhirne" >}}

dir

Generates a list of links to files within the site. Uses the “static/resources” directory as the base. Expects the following named args:

  • path: the path of to the directory
  • recursive: whether or not to expand sub directorys recursively
Example
{{< dir path="resources/drawings" recursive="true">}}

fig

Wraps content within a figure element. Generally this is used to display table figures. Expects the following named args:

  • caption: optional caption
  • class: optional css class list
  • exportable: if set true then provide an ‘export’ link. should only be used with tables.
  • id: optional id
  • link: optional link url for the figure
  • style: optional css styles

CSS Classes:

  • full-width - sets width to 100%
  • no-center - left aligns the figure
Example
{{< fig caption="A table" >}}
*a markdown table to go here...*
{{< /fig >}}`

img

Creates an image figure. Expects the following named args:

  • caption: optional caption
  • class: optional css class list
  • id: optional id.
  • link: optional link url for the image
  • src: may be either
    1. A URL
    2. A full path as “/a/full/path/img.png”
    3. An file name. This option assumes the file is within a dedicated directory for the document. For example, the images for document “guides/getting-started/introduction/index.md” would be expected to be located within “sguides/getting-started/introduction/".
  • style: optional css styles

CSS Classes:

  • full-width - sets width to 100%
  • no-center - left aligns the figure
Example
{{< img src="images/puppy.png" caption="A Puppy" >}}

pre

Wraps content within a pre element. Expects the following named args:

  • class: optional css class list
  • id: optional id
  • style: optional css styles

CSS Classes:

  • bordered - adds a border to the content
  • select-all - selects all text when content is clicked on
Example
{{< pre  >}}
*some text here...*
{{< /pre >}}

section

Wraps content within a section element. Expects the following named args:

  • class: optional css class list
  • id: optional id
  • style: optional css styles
  • title: optional title which will be wrapped within a header for the section.

CSS Classes:

  • bordered - adds a border to the content
  • callout - renders the section as a callout box
  • callout red, callout green - sets the left border color of the callout
  • checklist - uses checkboxes for any lists within the section
Example
{{< section  >}}
*some text here...*
{{< /section >}}

urlfor

Returns a URL from site data. Expects 1-3 arguments which represent the site data keys. See Site Links for a list of keys for site data.

Example
{{< urlfor vmware my >}}

Site Data

Authors

This data provides extra data for the author. Files should be created as yaml data with naming in the format of firstname-lastname.yml. Fields are as follows:

  • name - author name
  • github - an optional link to the author’s github user page
  • link - an optional personal link for the author

URLs

URL data is used as a means of centrally managing commonly referenced URLs within the site. Files are generally broken up per category (such as VMware Cloud). Files are yaml formatted dictionaries of the following objects:

  • name - display name for the link
  • url - link URL
  • topics - a dictionary of links within this topic

Topic entries should have a unique key and contain the following fields:

  • name - display name for the link
  • url - link URL

This data is accessed using the urlfor shortcode using the filename (minus the file extension) and the url key as the parameters.

Printable Views

All pages within the site are printable; however, it is sometimes useful to print multiple pages as a single document. For guides, there exists the concept of a printable view whose purpose is to aggregate child pages into a single page. This page is designed to enable printing.

To create a printable view, simply create a file called “printable.md” within a branch directory with the following content:

---
title: "Printable View"
subheading: "an optional sub-heading"
weight: 100
type: guides
layout: printable
---

A single-page view of this portion of the guide which is designed to enable easy printing. It is not intended for daily use.

See the noprintable Font Matter option for a means of excluding leaf or branch pages from a printable view.
